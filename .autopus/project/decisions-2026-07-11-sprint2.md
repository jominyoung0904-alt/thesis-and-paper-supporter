# Sprint 2 확정 사항 (2026-07-11, 실사용 피드백 기반)

> SPEC-TSA-002의 단일 소스. Sprint 1(SPEC-TSA-001) 실사용 테스트에서 나온 사용자 피드백과 추가 승인 기능.

## 배경

Sprint 1 앱이 실기기에서 종단 동작 확인됨(마법사→Gemini→딥리서치(OpenAlex+네이버 전문자료)→참고문헌 링크). 사용자는 연구를 동시에 3~4개 병행하며, 현재 앱은 단일 프로젝트(data/projects/default)라 연구별 분리가 불가.

## 중심 설계: 프로젝트 관리

- 상단에 **프로젝트 전환 UI** (목록/만들기/이름 변경/전환). 삭제는 데이터 소중함을 고려해 보관(숨김) 방식 우선.
- 모든 영속 데이터를 `data/projects/{projectId}/` 아래로 분리: memory.json(기존), library.json(보관함), chats/(대화), research/(리서치 이력), gate/(서론 점검 기록).
- 기존 default 프로젝트 데이터는 첫 실행 시 자동 마이그레이션(이름: "내 연구 1" 등).
- MemoryStore·ConversationManager 등 서비스가 "현재 프로젝트" 전환에 반응해 재조립돼야 함 (handlers의 조립 구조 확장).

## 기능 요구 (사용자 피드백 ④~⑦ + 승인된 추천 4종)

1. **문헌 보관함 (피드백④)**: 리서치 결과의 참고문헌·관련 문헌 각 항목에 저장 버튼(체크) → 현재 프로젝트 보관함에 저장(PaperMetadata 전체 + 저장일 + 출처 리서치). 보관함 화면(새 탭 또는 프로젝트 화면): 목록, 메모 한 줄, 삭제, 링크 열기.
2. **리서치→회의 연계 (피드백⑤)**: 리서치 완료 화면에 "이 결과로 회의하기" 버튼 → 채팅 모드로 전환되며 리포트 요약+참고문헌 목록이 대화 컨텍스트(summary 턴)로 주입된 새 대화 시작.
3. **대화 기록 영속 (피드백⑥)**: 대화는 프로젝트별 자동 저장(턴마다 또는 주기). 대화 목록 UI(제목=첫 질문 요약, 날짜), 선택 시 이어하기(restoreHistory), 새 대화 시작 버튼. ChatGPT식 경험이 목표.
4. **서론 점검 기록 (피드백⑦)**: 점검 실행 시 원고 텍스트+GateResult+시각을 프로젝트별 저장. 이력 목록에서 과거 점검 다시 보기.
5. **서지 복사 (추천)**: 보관함에서 선택 문헌들을 APA 형식 텍스트로 클립보드 복사(국문/영문 혼용 대응은 단순 규칙). 형식은 확장 가능하게(후속: BibTeX).
6. **리서치 이력 (추천)**: 딥리서치 결과(질문·리포트·citedPapers·relatedPapers)를 프로젝트별 자동 저장, 목록에서 다시 열람(재검색 낭비 방지). 열람 화면에서도 보관함 저장·회의하기 가능.
7. **모의 심사 (T21)**: 원고 텍스트 → Reviewer 2 관점 예상 질문·약점 (단일 모델 역할극, 기존 설계 결정 3). 결과도 프로젝트별 저장(4와 동일 패턴).
8. **문장 다듬기 (T20)**: 문단 입력 → 학술 문체 교정(국문/영문), 변경 사유 간단 표시.
9. **자동 백업 (T26)**: 앱 종료 시 data/ 백업 zip 생성, 최근 5개 보관(설계 결정 6).
10. **딥리서치 체크포인트 (T16)**: 진행 중 중단 시 재실행에서 이어하기(FR-RES-007/008).

## UX 원칙 (기존 유지 + 추가)

- 프로젝트 전환은 한 번의 클릭으로, 전환 시 현재 화면 유지(가능한 한).
- 저장은 "자동"이 기본(사용자가 저장을 잊어도 데이터가 남는다), 명시적 삭제만 수동.
- 새 탭이 늘어나므로 탭 구성 재검토 허용: 💬 대화 / 🔍 리서치(이력 포함) / 📚 보관함 / ✍️ 글쓰기(서론 점검+다듬기+모의 심사 통합) / ⚙️ 설정 — 구현 시 판단.

## 제약 (기존 전부 유지)

파일 300줄, 코드 주석 영어/UI 한국어 존댓말, 실패=결과값, 서지 결정론(FR-RES-005), 경로는 paths.ts 경유, preload 상대 import 금지, IPC 런타임 가드, LLM 무료 모드 rate limit 준수(회의 컨텍스트 주입 시 토큰 예산 주의 — 컴팩션 재사용).

## 구현 결과 (Sprint 2 완료, 2026-07-11)

Wave 0~8(T36~T63) 전부 완료. 구현 과정에서 아래 지점은 위 결정과 다르게 확정되었거나, 위 결정에는 없던 세부 사항이 새로 확정되었다.

1. **자동 백업 타이밍 — 종료 아닌 시작**: 설계 결정 5(zip 방식)는 유지했으나 실행 시점은 SPEC-TSA-001 결정 6이 전제한 "세션 종료 시"에서 **다음 세션 시작 시(창 생성 3초 후, fire-and-forget)**로 바꿨다. Windows `before-quit` 훅은 OS가 프로세스를 강제 종료하는 경로(창 닫기, taskkill, 강제 종료) 앞에서 비동기 정리가 끝나기 전에 죽을 수 있어 신뢰할 수 없고, 동기 핸들러는 종료를 지연시키다 강제 종료로 zip이 손상될 위험이 있다. 상세 근거는 `app/src/main/backup/sessionBackup.ts` 모듈 문서 참조.
2. **리서치 이력 상한 50 / 서론 점검 기록 상한 30 / 모의 심사 기록 상한 30**: "보류" 절이 리서치 이력을 무제한으로 제안했으나, 다른 프로젝트별 스토어(대화 세션 제외)와 동일한 "오래된 것부터 자동 정리" 패턴을 리서치 이력·점검 기록·모의 심사 3곳에도 적용해 디스크 사용량을 예측 가능하게 통제했다. 각 스토어의 `MAX_RECORDS` 상수로 구현.
3. **회의하기 핸드오프 컨텍스트 캡 8000자**: 설계 결정 6(요약+상위 N편 인용)의 구체적 상한을 `TOTAL_CHAR_HARD_CAP = 8000`(문자 기준, `compaction.ts`의 2.5자/토큰 근사치 재사용)으로 확정. `core/chat/researchHandoff.ts`에 하드 캡으로 구현.
4. **레코드 id UUID 가드 불변식 (Phase 4 리뷰 H1)**: 원 설계에 없던 항목. 리서치이력·대화세션·점검기록·모의심사 4개 스토어가 `{id}.json` 경로를 조립하는 지점에서 `core/persistence/recordId.ts::isSafeRecordId`로 핸들러·스토어 양쪽을 검증하도록 신설했다 — IPC 핸들러 가드만으로는, 향후 이 스토어를 직접 호출하는 새 경로가 추가될 경우 방어가 무력화될 수 있어 스토어 자체에도 두 번째 독립 방어선을 둔 것. ARCHITECTURE.md 핵심 불변식 9번으로 승격.
5. **장시간 핸들러의 프로젝트 경로 스냅샷 (Phase 4 리뷰 H2)**: 원 설계에 없던 항목. `research:run`·`quality-gate:run`처럼 실행 시간이 긴 핸들러는 진입 시점에 프로젝트 스코프 경로(researchDir/checkpointFile/gateDir)를 1회 스냅샷하고 실행 중에는 액세서를 재호출하지 않도록 확정 — 실행 도중 사용자가 프로젝트를 전환해도 교차 프로젝트 오기록·체크포인트 오염이 발생하지 않는다. ARCHITECTURE.md 핵심 불변식 10번으로 승격.
6. **M3/M4 의도적 백로그**: Phase 4 리뷰·보안 감사에서 지적된 M3(core 레이어의 예외/결과값 반환 규약이 모듈마다 다름 — 일부는 throw, 일부는 Result 객체)와 M4(`atomicWriteJson`류 원자적 저장 로직이 4개 스토어에 중복 구현됨)는 이번 스프린트에서 고치지 않았다. 사유: 둘 다 동작 결함이 아닌 유지보수성 이슈이고, 스프린트 마지막 단계에서 다수 스토어 파일을 동시에 손대는 리팩터는 이미 832개 테스트가 통과한 상태에서 회귀 위험 대비 얻는 미용적 이득이 작다고 판단했다. 다음 스프린트로 명시적으로 이월(Directive, d6ab09f 커밋 트레일러).
