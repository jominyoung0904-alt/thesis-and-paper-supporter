## 외부 API 연동 시 확인 필요 사항

### KCI OPEN API (공공데이터포털 경유)
- 공공데이터포털을 통한 인증 방식(서비스 키 발급, 활용신청 승인 소요 기간)을 구현 착수 전 실제로 확인해야 한다. decisions 문서는 "자동승인 가능성 높음"으로 기재했으나 실제 승인 SLA는 포털 문서로 재검증이 필요하다.
- 요청/응답 포맷(XML 또는 JSON), 페이지네이션, 초당/일당 호출 한도를 확인해 NFR-ACAPI-001(내장 공용 키) 설계 시 공용 키의 호출 한도 분배 정책에 반영한다.
- 제목/초록/저자/인용수 필드가 응답에 모두 포함되는지 확인 필요 - 일부 필드(인용수 등)가 KCI에서 제공되지 않을 경우 FR-RES-002의 수집 항목을 API별로 조정해야 한다.

### ScienceON API (KISTI)
- 토큰 발급/갱신 방식: decisions 문서에 명시된 대로 토큰이 2시간 만료되므로, FR-RES-009/NFR-ACAPI-003 구현 시 토큰 만료 임박(예: 만료 10분 전) 감지 후 자동 갱신하는 정확한 갱신 엔드포인트와 인증 파라미터를 KISTI 공식 문서에서 확인해야 한다.
- 개인 키 발급이 "수동 승인, 수일 소요"로 안내되어 있으므로, 실제 승인 절차(신청 화면, 필요 서류)를 스크린샷 마법사(NFR-ACAPI-002) 제작 전에 직접 확인해야 한다.
- 학위논문 포함 여부, 원문(PDF) 접근 가능 여부(현재는 불가로 전제)를 재확인하여 FR-RES-006의 안내 문구가 정확한지 검증한다.

### Semantic Scholar API
- 무료 API의 호출 한도(초당 요청 수) 및 API 키 미등록 시 제한 수준을 확인해야 한다. 등록 키 없이도 사용 가능한지, 등록 시 한도가 상향되는지에 따라 NFR-ACAPI-001의 내장 공용 키 전략에 Semantic Scholar도 포함할지 결정한다.
- 응답 필드(title, abstract, authors, citationCount, externalIds 등)가 FR-RES-002 요구사항(제목/초록/저자/인용수)과 1:1 대응되는지 확인한다.

### LLM 3사 어댑터 (Claude, Gemini, OpenAI)
- 3사의 API 인터페이스(요청 스키마, 스트리밍 방식, 토큰 사용량 응답 필드, 에러 코드 체계)가 서로 다르므로 NFR-LLM-001의 공통 어댑터 인터페이스는 최소공배수가 아니라 "요청(prompt, context, model), 응답(text, usage), 에러(코드, 메시지)"의 정규화된 형태로 설계해야 한다.
- 각 프로바이더의 429/인증/크레딧 소진 오류의 실제 HTTP 상태 코드와 에러 바디 포맷을 확인하여 NFR-LLM-004(한국어 오류 번역)의 매핑 테이블을 프로바이더별로 구성한다.
- 프롬프트 캐싱(FR-MEM-005의 캐시 친화적 직렬화와 연결) 지원 여부와 방식이 프로바이더마다 다르므로(예: 고정 프리픽스 캐싱), 실제 지원 여부를 프로바이더 문서에서 재확인한 뒤 캐시 적중률을 극대화하는 프롬프트 구조를 설계한다.

## 설계 결정 근거 보강

spec.md의 설계 결정 1~6번은 PRD 10절의 미해결 질문에 대한 기본값 제안이며, 근거는 다음 리서치 결과와 연결된다.

- 저장 포맷(JSON): 위 "그린필드" 특성상 성능 벤치마크 데이터가 없으므로, 1단계 데이터 규모(사용자 1인의 연구노트 수준)에서는 SQLite 도입에 따른 운영 복잡도(스키마 마이그레이션, 바이너리 포맷의 비가독성)가 이점보다 크다고 판단.
- ScienceON 승인 지연 UX: 실제 승인 SLA를 아직 확인하지 못했으므로(위 "확인 필요 사항" 참고), 배너 문구의 구체적 소요 기간("통상 수일")은 실제 확인 후 갱신이 필요하다. 이 부분은 원격 설정(공지사항)으로 갱신 가능하게 설계해 하드코딩을 피한다.

미해결 질문 7번(원격 설정 검증 실패 시 사용자 노출 여부)은 spec.md에서 보류로 남겨두었으며, 사용자 확인 후 NFR-CFG-004를 확정 요구사항으로 전환해야 한다.

## 국내 API 전환 결정(2026-07-11)

### 실측 결과: KCI/ScienceON 직접 호출 불가

배포된 데스크톱 앱 환경에서 실제 호출을 시도한 결과, 두 국내 API 모두 위 "확인 필요 사항" 절에서 우려했던 수준을 넘어 원천적으로 배포 앱에서 사용할 수 없음이 확인되었다.

- **KCI OPEN API(공공데이터포털 경유)**: 서비스 키 발급 시 등록한 **IP 단위**로 호출이 제한된다. 사용자마다 공인 IP가 다른 포터블 데스크톱 앱 특성상, 앱에 키를 내장(NFR-ACAPI-001의 "공용 키" 전략)해도 개발자의 IP에서만 동작하고 실제 사용자 환경에서는 인증에 실패한다.
- **ScienceON API(KISTI)**: 개인 키 발급 시 등록한 **MAC 주소 단위**로 호출이 제한된다. IP보다도 더 엄격한 제약으로, 사용자 PC마다 MAC이 다른 이상 공용 키 배포 자체가 불가능하다.
- 결론: `BUNDLED_ACADEMIC_KEYS`(bundledKeys.ts)를 통한 공용 키 배포 전략(NFR-ACAPI-001)은 KCI/ScienceON에 대해 **폐기**한다. 두 클라이언트 코드(kciClient.ts, scienceOnClient.ts)와 목업 데이터는 사용자가 직접 개인 키를 발급받아 로컬에 등록했을 때만(개인용 고급 옵션) 사용되는 경로로 남긴다.

### 전환 결정: OpenAlex(키리스) 도입

OpenAlex(https://api.openalex.org)를 국내 학술 검색의 기본 소스로 전환한다.

- **실증 확인**: 키 등록이나 IP/MAC 화이트리스트 없이 누구나 호출 가능. 한국어 검색어(`search=` 파라미터)로 조회 시 KCI에 DOI로 등재된 국내 학술지 논문이 결과에 포함됨을 실제 호출로 확인했다.
- **응답 스키마**: `GET /works?search=&per-page=&select=...` → `{ results: [...] }`. 초록은 저작권상의 이유로 평문이 아니라 `abstract_inverted_index`(단어→위치 배열의 역인덱스)로 제공되므로, 클라이언트가 위치 순서대로 재조립해야 한다(`openAlexClient.ts`의 `reconstructAbstract`).
- **아키텍처 변경**: 애초 검토했던 "중계 서버(사용자 대신 앱 개발자 서버가 KCI/ScienceON을 호출)" 계획은 운영 비용·가용성 부담이 크고, OpenAlex가 키리스로 동일한 목적(국내 학술지 검색)을 충족하므로 **폐기**한다.
- `academicClients.ts`: OpenAlex는 Semantic Scholar와 함께 항상 실모드로 포함한다. 국문 검색어 배정(`pipeline.ts`의 `termsForSource`)은 "semanticscholar만 영문, 나머지는 국문" 규칙이 그대로 적용되므로 코드 변경이 불필요하다 — OpenAlex가 자동으로 국문 검색을 담당한다.

### 한계 (사용자 안내 문구/향후 스프린트 반영 필요)

- **학위논문 미포함**: OpenAlex는 학술지·프리프린트·컨퍼런스 논문 위주이며, 국내 학위논문(RISS 등재분)은 색인 대상이 아니다.
- **비-DOI 국내 학술지 미포함**: DOI가 없는 일부 국내 학술지(특히 등재후보지·비영리 소규모 학회지)는 OpenAlex에 없을 수 있다.
- **국내 논문 초록 결측**: `abstract_inverted_index`가 아예 없는 국내 논문 레코드가 영어 논문보다 흔하게 관찰됨 — 리포트 스크리닝(FR-RES-003) 시 초록 없는 논문의 관련성 판단 정확도가 낮아질 수 있다.

### Sprint 2 선택 항목: Google Programmable Search(RISS 보조)

위 한계(특히 학위논문 미포함) 보완을 위해, RISS(학술연구정보서비스)를 대상으로 한 Google Programmable Search Engine 연동을 1단계 범위 밖의 선택 항목으로 남겨둔다. Google Programmable Search는 무료 티터 호출 한도(일 100회)가 낮아 기본 경로로 채택하기 어렵고, RISS 자체가 공식 오픈 API를 제공하지 않아 검색 결과 스크레이핑에 준하는 통합이 필요하므로 별도 SPEC으로 분리해 검토한다.
