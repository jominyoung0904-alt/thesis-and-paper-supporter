## 1. 개요 및 배경

**논문 작성 서포터**는 한국 대학원생(사회과학·공학 등 분야 무관)을 위한 로컬 데스크톱 앱이다. 대학원생의 논문 작성 전 과정을 하나의 "연구 프로젝트 메모리"로 관통하며 보조하는 것이 핵심 컨셉이다.

타깃 사용자는 명시적으로 컴퓨터 비숙련자다. "조금만 복잡해도 포기하는 사용자"를 UX 설계 기준으로 삼는다. 이는 배포 형태(설치 프로그램 없는 포터블 zip), 상호작용 모델(채팅 하나로 모든 기능 접근), 오류 메시지(일상어 번역) 등 제품 전반의 설계 원칙을 규정한다.

기존 도구(Elicit, SciSpace, scite, Zotero)는 문헌 탐색·인용 관리 등 개별 기능에 특화되어 있으나, 연구 전 과정을 관통하는 맥락을 유지하지 못한다. 본 제품은 다음 세 가지 차별화 포인트로 이 공백을 메운다.

1. 연구 전 과정을 관통하는 프로젝트 메모리 - 파편화된 기존 도구들이 채우지 못하는 빈자리
2. 연구 결정 이력 추적 - 연구노트·심사 대응 자료로 재활용 가능
3. 인용 무결성 검증을 기본 내장한 집필 보조 - 가짜 인용을 구조적으로 차단

논문 언어는 국문·영문 모두 지원한다. 아키텍처는 전 과정(주제선정 -> 문헌조사 -> 연구설계 -> 데이터 -> 집필 -> 검토 -> 심사대응) 통합을 목표로 설계하되, 구현은 단계적이다. 본 PRD는 1단계 구현 범위를 다룬다.

## 2. 문제 정의

대학원생, 특히 컴퓨터 비숙련자인 대학원생이 논문을 작성하는 과정에서 겪는 문제는 다음과 같다.

- 맥락 단절: 문헌 탐색, 집필, 검토 도구가 분리되어 있어 "왜 이 방법론을 선택했는지", "왜 이 문헌을 제외했는지" 같은 연구 결정의 근거가 도구 간에 휘발된다. 매 세션이 첫 날처럼 반복된다.
- 가짜 인용 위험: LLM 기반 집필 보조 도구는 실재하지 않는 논문을 인용문으로 생성하는 환각 문제가 있다. 인용·서지정보를 LLM 생성에만 의존하면 검증되지 않은 인용이 논문에 삽입될 위험이 크다.
- 진입 장벽: 기존 AI 리서치 도구는 설치·설정·API 키 발급 절차가 복잡해 컴퓨터 비숙련자가 중도 포기하기 쉽다. 사내망·개인 PC 등 관리자 권한이 없는 환경에서 설치 자체가 막히는 경우도 있다.
- 비용 불확실성: LLM 기반 도구를 유료로 쓸 경우 예상치 못한 과금이 발생할 수 있어, 학생 신분의 사용자에게는 진입 장벽이자 이탈 요인이 된다.
- 심사 대응 준비 부족: 논문 심사 과정에서 예상 질문·비판에 대한 사전 대응 자료를 준비하기 어렵고, 연구 결정 근거를 사후에 재구성하는 데 시간이 많이 든다.

## 3. 목표 및 성공 지표

### 목표

1. 컴퓨터 비숙련자도 zip 압축 해제 후 5분 이내에 첫 검색 결과를 받아볼 수 있게 한다.
2. 연구 프로젝트 메모리를 모든 기능(딥리서치, 집필, 검토)의 공유 컨텍스트로 주입하여 맥락 단절을 제거한다.
3. 학술 API(KCI·ScienceON·Semantic Scholar) 직접 조회를 통해 인용 무결성을 구조적으로 보장한다(LLM이 서지정보를 생성하지 않고, 검증만 수행).
4. 무료 모드(Gemini Flash 무료 티어)만으로도 핵심 기능(딥리서치, 집필 보조)이 동작하게 하여 비용 부담 없이 제품을 체험할 수 있게 한다.

### 성공 지표 (측정 가능)

| 지표 | 목표값 | 측정 방법 |
|---|---|---|
| 최초 실행 성공률 | zip 해제 후 exe 실행 -> 설정 마법사 완료까지 성공률 90% 이상 | 앱 내 익명 이벤트 로그(설정 마법사 완료 여부) |
| 딥리서치 응답 시간 | 무료 모드 기준 1회 조회 2~3분 이내 완료(rate limit 준수 하에) | 앱 내 처리 시간 로깅 |
| 가짜 인용 발생률 | 0% (모든 인용은 KCI/ScienceON/Semantic Scholar 응답에서만 생성, LLM이 서지정보 자체를 생성한 인용은 구조적으로 불가능) | 코드 검증: 인용 필드가 API 응답 ID와 1:1 매핑되는지 정적 검증 |
| SmartScreen 이탈률 | 안내 HTML 도입 후 최초 실행 중도 포기율 10% 미만 | 설정 마법사 진입률 대비 완료율 |
| 무료 모드 완주율 | 무료 모드 사용자 중 딥리서치 1회 이상 완료 비율 70% 이상 | 앱 내 이벤트 로그 |
| 데이터 유실 사고 | 0건 (자동 백업 동작 기준) | 백업 파일 생성 로그, 사용자 리포트 |
| 품질 게이트 통과 없이 섹션 완료 처리된 비율 | 0% (품질 게이트 미통과 시 완료 처리 차단) | 앱 내 상태 전이 로그 |

## 4. 사용자 페르소나 및 시나리오

### 페르소나: 김OO (사회과학 계열 석사 2년차)

- 컴퓨터 사용은 워드프로세서·웹서핑 수준. 터미널, 환경변수, API 키 같은 개념에 익숙하지 않음.
- 지도교수와의 미팅 전 선행연구를 정리해야 하지만 시간이 부족하고, 어떤 검색어로 찾아야 할지도 막막함.
- 논문 서론을 쓰다가 "이 주장에 인용이 있어야 하나?" 같은 판단을 스스로 내리기 어려움.
- 컴퓨터를 학교/집 두 곳에서 사용하며, 두 PC에 설치 프로그램을 깔고 싶어 하지 않음(관리자 권한 이슈 우려).

### 핵심 시나리오

1. 첫 실행: zip을 다운로드해 바탕화면에 압축 해제 -> exe 더블클릭 -> SmartScreen 경고가 뜨자 안내 HTML을 참고해 "추가 정보 -> 실행" -> 설정 마법사에서 무료 모드(Gemini) 선택 -> API 키 발급 안내를 따라 Google AI Studio에서 키 발급 -> 첫 화면 진입.
2. 딥리서치: 채팅창에 "국내 대학생 SNS 중독 관련 연구 있어?"라고 입력 -> 앱이 국문+영문 검색어를 자동 생성해 KCI·ScienceON·Semantic Scholar를 병렬 조회 -> 관련도 순으로 스크리닝된 종합 리포트(실제 링크 포함)를 2~3분 후 받음.
3. 집필: 서론 섹션을 작성하며 앱이 실시간으로 "연구 갭이 명시되지 않았어요", "이 주장에 인용이 필요해요" 같은 품질 게이트 체크를 제공 -> 인용을 클릭하면 원본 링크로 이동해 실재 여부 확인 가능.
4. 모의 심사: 초고 완성 후 Reviewer 2 모의 심사를 요청 -> 앱이 적대적 관점에서 예상 질문·약점을 제시 -> 사용자는 이를 심사 대응 자료로 활용.
5. 이동/백업: 다른 PC로 작업을 옮길 때 폴더 전체를 복사 -> data/ 폴더가 그대로 포함되어 있어 즉시 이어서 작업 가능.

## 5. 기능 요구사항 (1단계 3개 모듈)

### 5.1 연구 프로젝트 메모리 코어

연구 프로젝트 메모리는 다른 두 모듈(딥리서치, 집필·검토)이 공유하는 컨텍스트 저장소이며, 1단계의 기반 모듈이다.

- 저장 대상:
  - 연구 질문(research question)
  - 가설
  - 용어 정의(연구에서 사용하는 핵심 개념의 조작적 정의)
  - 연구 결정 이력 - 각 결정에 대해 "무엇을 결정했는가"와 "왜 그렇게 결정했는가"(예: 왜 이 방법론을 채택했는가, 왜 이 문헌을 제외했는가)를 함께 기록
  - 지도교수 피드백 - 미팅/이메일에서 받은 피드백을 구조화하여 저장, 추후 대응 여부 추적
- 공유 컨텍스트 주입: 딥리서치와 집필·검토 모듈이 LLM을 호출할 때, 프로젝트 메모리의 관련 항목이 자동으로 프롬프트 컨텍스트에 포함되어야 한다("매 세션이 첫 날" 문제 방지).
- 캐시 친화적 설계: 프로젝트 메모리는 LLM 호출 시 프리픽스로 주입되므로, 프롬프트 캐싱을 활용할 수 있는 구조로 직렬화한다(비용 절감 목적, 6절 NFR 참고).
- 결정 이력 활용: 연구 결정 이력은 연구노트 형태로 내보내기 가능해야 하며, 심사 대응 자료 작성 시 참조할 수 있어야 한다.
- 저장 위치: data/ 폴더 내 구조화된 형식(예: JSON/SQLite 등 - 저장 포맷은 설계 단계에서 결정)으로 저장하여 포터블성을 보장한다.

### 5.2 선행연구 딥리서치

- 입력: 사용자가 채팅으로 자연어 질문(예: "이런 연구 있어?")을 입력한다.
- 검색어 생성: LLM이 질문을 분석해 국문 및 영문 검색어를 각각 생성한다(다국어 학술 검색 대응).
- 병렬 메타데이터 검색: 생성된 검색어로 KCI OPEN API, ScienceON API, Semantic Scholar API를 병렬 조회한다. 조회 대상 메타데이터는 제목, 초록, 저자, 인용수를 포함한다.
- 관련도 스크리닝: 검색 결과를 연구 질문과의 관련도 기준으로 스크리닝한다(경량 모델 사용 - 6절 참고).
- 종합 리포트 생성: 스크리닝된 결과를 종합하여 리포트를 생성한다. 리포트 내 모든 인용에는 실제 원문 링크(또는 원문 접근 안내)가 첨부되어야 한다.
- 인용 무결성 보장: LLM은 리포트의 해석·요약만 담당하며, 서지정보(저자·제목·발행연도 등)는 학술 API 응답에서 결정론적으로 가져온다. LLM이 서지정보 자체를 생성하는 경로는 구조적으로 존재하지 않아야 한다.
- 체크포인트/재개: 딥리서치는 LLM 호출 10~15회, 입력 약 10만 토큰, 출력 약 8천 토큰 규모로 장시간 소요될 수 있다(8절 리스크 11번 참고). 중간 진행 상태를 체크포인트로 저장하여 중단 시 이어서 진행할 수 있어야 한다.
- 원문 접근 한계 안내: KCI·ScienceON·Semantic Scholar 모두 원문(PDF) 직접 접근을 제공하지 않으므로, 리포트는 초록 기반 판단 결과이며 원문은 "학교 도서관 계정으로 열람" 등 안내 문구와 함께 링크만 제공됨을 명시한다.

### 5.3 집필·검토 보조

- 섹션별 품질 게이트 체크리스트: 논문 섹션(예: 서론)마다 성공 기준을 체크리스트로 제공한다. 예: 서론 = 연구 갭 명시 + 기여 명시 + 모든 주장에 인용. 절차적 지시가 아니라 기준 충족 여부를 검증하는 방식으로 동작한다.
- 품질 게이트 미통과 시 처리: 체크리스트 기준을 만족하지 않으면 해당 섹션을 "완료"로 표시할 수 없도록 차단하거나 경고를 명확히 표시한다(3절 성공 지표 참고).
- 학술 문장 다듬기: 사용자가 작성한 문장을 학술적 문체로 다듬는 기능을 제공한다(국문·영문 모두 지원).
- 모의 심사(Reviewer 2 시뮬레이션): 다중 모델 Debate 패턴을 적용하여 적대적 관점에서 리뷰를 수행한다. 예상 질문·약점을 제시하여 심사 대응 준비를 돕는다.
- 인용 클릭 검증: 집필 중 삽입된 인용을 클릭하면 원본 링크로 이동해 실재 여부를 즉시 확인할 수 있어야 한다(딥리서치 모듈에서 확보한 링크를 재사용).
- 프로젝트 메모리 연동: 집필·검토 시 프로젝트 메모리(연구 질문, 가설, 용어 정의)를 컨텍스트로 활용하여 일관성을 검증한다.

## 6. 비기능 요구사항

### 6.1 포터블 배포

- Electron 기반 데스크톱 앱, 포터블 zip으로 배포(zip 크기 150~250MB 허용). Tauri 대신 Electron을 채택한 이유는 WebView2 의존성 없이 모든 Windows 10/11 환경에서 동작을 보장하기 위함이다.
- 설치 프로그램 없음, 레지스트리·AppData 미사용, 관리자 권한 불필요. 압축 해제 후 exe 더블클릭으로 즉시 실행되어야 한다.
- 폴더 구조는 다음을 따른다.

```
논문서포터/
├── 논문서포터.exe
├── 처음이라면_읽어주세요.html   (SmartScreen 대처 스크린샷 포함)
├── app/                        (프로그램 본체, 업데이트 시 교체)
├── data/                       (사용자 데이터: 메모리, 라이브러리, 키)
└── config/settings.json        (메모장으로도 수정 가능한 설정)
```

- 원칙: "폴더 복사 = 백업", "폴더 이동 = 그대로 동작", "새 버전 = app/만 교체, data/ 보존". 업데이트 절차는 이 원칙을 위배하지 않아야 한다.

### 6.2 비숙련자 UX

- 채팅 하나로 모든 기능(딥리서치, 집필 보조, 모의 심사 등)에 접근 가능해야 한다. 별도 메뉴 탐색을 강제하지 않는다.
- 설정 화면은 최소화한다. 꼭 필요한 설정(제공자 선택, API 키, 학술 키 등록)만 노출한다.
- 모든 안내 메시지와 에러 메시지는 한국어 일상어로 번역되어야 한다. 기술 용어(예: 429 오류, rate limit exceeded)를 그대로 노출하지 않는다(예: "오늘 무료 사용량을 다 썼어요. 내일 오후 5시에 다시 채워져요").
- 최초 실행 설정 마법사는 LLM 제공자 선택과 API 키 입력을 포함하며, 키 발급 절차를 스크린샷 기반 단계별 안내로 제공한다.

### 6.3 프라이버시

- API 키는 로컬 암호화 저장하며, 설정 화면에서 언제든 교체할 수 있어야 한다.
- 키 파일은 사용자 데이터와 분리 저장하고, "다른 사람에게 줄 때는 키 제외 내보내기" 기능을 제공한다(8절 리스크 4번).
- 기능별 품질·프라이버시 안내 문구는 하드코딩하지 않고 원격 설정(endpoints.json 등)에서 로드한다.
  - 무료 모드 안내: "결과 품질은 같지만 속도가 느리고, 입력 내용이 구글 AI 학습에 쓰일 수 있어 작성 중인 원고 업로드는 권하지 않아요"
  - 유료 모드 안내: "Pro급 모델로 리포트·문장 다듬기·모의 심사 품질 향상 + 속도 제한 없음 + 입력 내용 학습 미사용"
  - 각 기능 화면에 현재 모드(무료/유료) 배지를 표시한다.

### 6.4 성능 / Rate Limit

- 무료 모드: Gemini Flash 무료 티어 기준 분당 10~15회 제한을 준수하는 rate limiter를 앱에 내장한다. 딥리서치 1회 처리 시간은 2~3분 수준으로 안내한다.
- 유료 모드: 작업 복잡도별 모델 자동 배치를 적용한다. 스크리닝은 경량 모델(예: Gemini Flash, Claude Haiku, GPT mini급), 리포트 종합·모의 심사는 상위 모델(예: Claude Sonnet, GPT-5.4급)을 사용한다.
- LLM 호출부는 Claude/Gemini/OpenAI 3사를 지원하는 어댑터 계층으로 추상화하여, 제공자 전환이 코드 변경 없이 설정만으로 가능해야 한다.
- 월 사용액을 실시간으로 표시하고 상한 알림 기능을 제공한다. 요금 안내표는 원격 설정으로 갱신 가능해야 한다(2026-07 기준 참고값: Gemini Flash 절약형 월 2달러 내외 / Claude Haiku 5달러 내외 / OpenAI mini 4달러 내외 / 균형형 8~15달러).
- 원격 설정(GitHub Pages 정적 호스팅 endpoints.json)은 앱 시작 시 확인하고, 실패 시 로컬 config/settings.json 값으로 폴백한다. 원격 설정 관리 대상: API 주소, 모델명(단종 대응), 요금 안내표, 공지사항, 내장 학술 키.

## 7. 범위 제외 (2단계 이후)

다음 항목은 1단계 구현 범위에서 명시적으로 제외한다. 단, 아키텍처 설계 시에는 이들의 확장 가능성을 고려한다.

- PDF 라이브러리 RAG: 사용자가 보유한 PDF 문헌 라이브러리에 대한 검색 증강 생성(RAG) 기능은 2단계 이후로 연기한다.
- 문헌 스크리닝 파이프라인: 대규모 문헌 리뷰를 위한 체계적 스크리닝 파이프라인(예: PRISMA 방식 지원)은 1단계 범위 밖이다.
- Zotero 연동: 외부 레퍼런스 매니저(Zotero) 연동은 2단계 이후로 연기한다.
- 데이터 단계(수집·분석) 및 심사대응 단계 전용 기능은 아키텍처상 고려 대상이나, 1단계에서는 구현하지 않는다(1단계는 메모리 코어·딥리서치·집필·검토 3개 모듈에 한정).

## 10. 미해결 질문

- 프로젝트 메모리 저장 포맷: JSON 파일 기반인지 SQLite 등 경량 DB를 사용할지 결정되지 않음. 포터블성(폴더 복사=백업)과 조회 성능을 함께 고려해 설계 단계에서 확정 필요.
- 품질 게이트 체크리스트의 판정 주체: 서론 외 나머지 섹션(본론, 결론 등)의 체크리스트 기준이 아직 정의되지 않음. 1단계에서 서론 외 몇 개 섹션까지 지원할지 범위 확정 필요.
- 모의 심사(Reviewer 2)의 다중 모델 Debate 구현 방식: 단일 모델의 역할극(prompt 기반)으로 구현할지, 실제로 2개 이상의 다른 모델을 교차 호출하는 방식으로 구현할지 결정되지 않음. 비용(모델 2회 호출)과 품질 트레이드오프 검토 필요.
- ScienceON API 승인 지연 시 UX: 수동 승인(수일 소요)이 완료되기 전까지 사용자가 내장 공용 키로만 이용하는 기간의 안내 문구·한도 배분 정책이 미정.
- KCI/ScienceON 응답에 관련 문헌이 없을 때의 처리: Semantic Scholar 단독 결과만 제공할지, "국내 문헌 없음"을 명시할지 등 부분 실패 시 리포트 구성 방식 미정.
- 자동 백업 주기 및 보관 정책: "주기적 자동 백업"의 구체적 주기(예: 매 세션 종료 시 vs. N분 간격)와 백업본 보관 개수·삭제 정책이 결정되지 않음(MVP 이후 항목이나 설계 초기에 방향 필요).
- 원격 설정(endpoints.json) 검증 실패 시 사용자 노출 여부: 원격 설정 로드 실패를 조용히 로컬 폴백만 할지, 사용자에게도 알릴지 미정.
